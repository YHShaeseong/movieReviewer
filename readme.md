## 픽토스(Picktos) 팀 GitHub 협업 가이드 🚀

우리 프로젝트의 성공적인 완성을 위해 간단한 규칙을 정합니다.
Git이 익숙하지 않아도 이 순서만 따라 하면 됩니다!

### 1. 핵심 브랜치 3가지 (가장 중요!)

우리는 3가지 종류의 브랜치를 사용합니다.

- **🌳 `main`**
    - **절대 금지:** 이 브랜치에는 **절대 아무도 직접 수정(Push)하지 않습니다.**
    - **용도:** 오직 **완전히 완성된, 발표 가능한 최종 버전**의 코드만 보관하는 곳입니다. (팀장님이 `develop` 브랜치에서 합칠 거예요)
- **🌿 `develop`**
    - **중심 브랜치:** **우리가 개발하는 모든 코드의 중심**입니다.
    - **용도:** 각자 개발한 기능(`feature`)이 완성되면, 이 `develop` 브랜치에 합칩니다.
    - **규칙:** 내가 새로운 기능 개발을 시작할 땐, **항상 이 `develop` 브랜치에서 최신 코드를 받아** 시작합니다.
- **🌱 `feature/기능이름`**
    - **개인 작업실:** **모든 실제 개발 작업은 이 브랜치에서** 합니다.
    - **용도:** 각자 맡은 기능을 개발하는 브랜치입니다.
    - **예시:**
        - 팝업창 기능 담당 (이승윤 님) -> `feature/popup`
        - API 연동 담당 (유해성 님) -> `feature/api`
        - 헤더 구현 담당 (조유찬 님) -> `feature/header`
        - 디자인 담당 (송차언 님) →  feature/design

### 2. 개발 작업 순서 (이것만 외우세요!)

내가 오늘 '팝업창' 기능을 개발한다면, 이 순서대로만 하세요.

1. **[내 PC] `develop` 브랜치로 이동**
2. **[내 PC] `develop` 최신 버전으로 업데이트** (명령어: `git pull origin develop`)
    - *“다른 팀원이 작업한 최신 내용을 먼저 받는다”*
3. **[내 PC] 내 기능 브랜치 생성 및 이동** (명령어: `git checkout -b feature/popup`)
    - *“feature/popup 이라는 나만의 작업실을 만든다”*
4. **[내 PC] 열심히 코딩 & 수정 (무한 반복)**
5. **[GitHub] 내 작업실에 코드 올리기** (명령어: `git push origin feature/popup`)
    - *“내 작업 내용을 내 이름표가 붙은(feature/popup) GitHub에 백업한다”*
    - 작업이 끝나지 않았어도, **퇴근 전이나 중간 백업으로 자주 push** 하세요!

---

### 3. 간단한 커밋(Commit) 규칙

커밋은 '수정 사항 저장'입니다. 나중에 누가 봐도 알기 쉽게 적는 게 중요합니다.

- **형식:** `타입: 작업 내용`
- **예시:**
    - `feat: 로그인 팝업창 UI 완성` (feat = 새로운 기능 추가)
    - `fix: 팝업창 닫기 버튼 오류 수정` (fix = 버그 수정)
    - `style: 메인 페이지 CSS 디자인 변경` (style = 코드 변경 없는 디자인 수정)
    - `docs: README 파일에 사용법 추가` (docs = 문서 수정)

### 4. 핵심 사용 규칙 3가지 (약속!)

1. **절대 `main`과 `develop` 브랜치에 직접 Push(수정)하지 않습니다.**
    - 모든 작업은 **`feature/`** 브랜치(내 작업실)에서만 합니다.
2. **기능이 완성되면 `develop` 브랜치에 'Pull Request (PR)'를 보냅니다.**
    - 'Pull Request'는 "팀장님, 저 팝업창 기능 다 만들었어요! `develop` 브랜치에 합쳐주세요!" 라고 요청하는 기능입니다.
    - 팀장님이나 다른 팀원이 코드를 확인(리뷰)하고 `develop`에 합쳐줄 거예요.
3. **작업 시작 전에는 항상 `git pull`을 먼저 합니다.**
    - 다른 사람이 작업한 내용을 내 PC에 먼저 반영해야 나중에 코드가 꼬이지 않습니다.
